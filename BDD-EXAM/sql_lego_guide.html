⚡ Concept 1

Triggers

A trigger is a motion-sensor alarm inside the database. You set it once, and it automatically fires whenever a specific event happens — no one has to manually activate it.

🚪

The automatic door analogy

Imagine a supermarket door. You don't press a button — you walk toward it, and it opens automatically. A trigger is the same: you INSERT a row, and the trigger's code runs automatically. You never have to call it.

The magic words: NEW and OLD

🔴 OLD.column

The value before the change.

Only available in UPDATE and DELETE triggers.

  • OLD.estado → previous status
  • OLD.cantidad → quantity before
⇄
row
changes

🟢 NEW.column

The value being inserted or the new value after update.

Available in INSERT and UPDATE.

  • NEW.cantidad → quantity ordered
  • NEW.estado → new status value

BEFORE trigger

Fires before the row is saved. Used to validate or block bad data.

BEFORE INSERT ON DetallePedidos BEGIN IF NEW.cantidad <= 0 THEN SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '❌ Quantity must be > 0'; END IF; END;

SIGNAL = throw an error and stop everything, like a bouncer at a door.

AFTER trigger

Fires after the row is saved. Used to react to what just happened.

AFTER INSERT ON DetallePedidos BEGIN -- Row is already saved; now update stock UPDATE productos SET cantidadEnStock = cantidadEnStock - NEW.cantidad WHERE codigoProducto = NEW.codigoProducto; END;

NEW.codigoProducto reads the value from the row just inserted.

🔄 Concept 2

Cursors

SQL normally works on all rows at once. A cursor lets you work on rows one at a time, like a reading finger that moves down a list — pick up card, read it, do something, move to next card.

🃏

The flashcard deck analogy

Imagine a stack of flashcards — one per client. A cursor is your hand: pick up card 1, read the name and phone, show it on screen. Pick up card 2, do the same. Keep going until the deck is empty. Then put the deck away (CLOSE).

1
DECLARE variables — create empty boxes to hold one row's data at a time.
DECLARE nombre_cliente VARCHAR(50);
2
DECLARE CURSOR FOR SELECT — tell MySQL which rows you want to loop through.
DECLARE cur CURSOR FOR SELECT nombreCliente, telefono FROM Clientes;
3
DECLARE CONTINUE HANDLER FOR NOT FOUND — when cursor runs out of rows, set a flag.
DECLARE CONTINUE HANDLER FOR NOT FOUND SET terminado = 1;
4
OPEN — point the cursor at the first row.
OPEN cur;
5
LOOP → FETCH — grab next row into your variables, do your work, check the "done" flag.
FETCH cur INTO nombre_cliente, telefono_cliente;
6
CLOSE — always clean up when done.
CLOSE cur;

Two loop styles — same result, different shape

LOOP / LEAVE style

Check the flag at the top of the loop (exercises 4–7)

lbl: LOOP FETCH cur INTO v; IF done=1 THEN LEAVE lbl; END IF; -- do stuff with v END LOOP;

REPEAT / UNTIL style

Check the flag at the bottom of the loop (exercise 8)

REPEAT FETCH cur INTO v; IF NOT done THEN -- do stuff with v END IF; UNTIL done END REPEAT;
📦 Concept 3

Stored Procedures

A procedure is a named recipe saved inside the database. Write it once, call it anytime. It can take inputs, run logic, use cursors, and print results.

🍪

The recipe card analogy

You write a cookie recipe once and put it in a drawer. Next time you want cookies, you say "make cookies!" — you don't rewrite the recipe. A stored procedure is a recipe card saved inside MySQL. CALL make_cookies() bakes them.

⚠️ Why DELIMITER // ? The most confusing part

MySQL uses ; to know when a statement ends. But a procedure body contains many semicolons. Without changing the delimiter, MySQL would read the first ; and think the procedure is done — it would crash mid-way.

❌ Without DELIMITER — MySQL gets confused CREATE PROCEDURE p() BEGIN SELECT * FROM Clientes; ← MySQL stops here SELECT * FROM Pedidos; ← never reached END;
✅ With DELIMITER // — MySQL reads it all DELIMITER // CREATE PROCEDURE p() BEGIN SELECT * FROM Clientes; ← fine, // not seen yet SELECT * FROM Pedidos; ← also fine END; ← still going... // ← NOW MySQL knows it's done DELIMITER ;
🧱 Full procedure skeleton — every procedure follows this shape
Procedure DELIMITER // CREATE PROCEDURE name_of_procedure( IN input_param VARCHAR(50) -- IN = caller gives you a value ) BEGIN -- ① DECLARE all variables FIRST (always at the top) DECLARE done INT DEFAULT 0; DECLARE v_name VARCHAR(50); -- ② DECLARE cursor (after variables, before handler) DECLARE cur CURSOR FOR SELECT nombreCliente FROM Clientes WHERE ciudad = input_param; -- ③ DECLARE handler (must come after cursor) DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = 1; -- ④ OPEN cursor OPEN cur; -- ⑤ LOOP through rows my_loop: LOOP FETCH cur INTO v_name; IF done = 1 THEN LEAVE my_loop; END IF; SELECT v_name AS Cliente; END LOOP; -- ⑥ CLOSE cursor CLOSE cur; END; // DELIMITER ; -- Call it like this: CALL name_of_procedure('Madrid');
🚨 Critical rule: DECLARE order matters!
In MySQL, inside BEGIN...END you MUST declare in this exact order:
1. Variables (DECLARE x INT) → 2. Cursors (DECLARE cur CURSOR FOR) → 3. Handlers (DECLARE CONTINUE HANDLER)
If you put the handler before the cursor, MySQL will throw an error.

🧮 Concept 4

Functions

A function is like a calculator button. Give it an input, get exactly one answer back. Unlike a procedure, you use a function directly inside a SELECT statement.

🧮

The calculator analogy

Press total_pedido(5) — calculator shows €47.50. That's it. One number back. You can use it anywhere you'd write a number: in a WHERE clause, in a SELECT, in another calculation.

Function vs Procedure — the key difference

🧮 FUNCTION 📦 PROCEDURE
ReturnsExactly ONE valueMany rows / nothing
Used withSELECT total_pedido(1)CALL total_por_cliente()
Can print rowsNo — only RETURNYes — SELECT inside
Can use cursorTechnically yes, rarelyYes — common pattern
Has RETURNS typeYes — must declareNo return type
🧱 The total_pedido function — fully annotated
Function DELIMITER // CREATE FUNCTION total_pedido(cod_pedido INT) ← input parameter RETURNS DECIMAL(10,2) ← declares the TYPE of value it gives back NO DETERMINISTIC ← result may change if data changes READS SQL DATA ← tells MySQL: this function reads tables BEGIN DECLARE total DECIMAL(10,2); ← box to hold the result SELECT SUM(cantidad * precioUnidad) INTO total ← saves query result into the variable FROM DetallePedidos WHERE codigoPedido = cod_pedido; RETURN IFNULL(total, 0); ← RETURN sends the value back to the caller ← IFNULL: if no rows matched, return 0 not NULL END; // DELIMITER ; -- Call it in a SELECT — like a built-in formula: SELECT total_pedido(1); → returns e.g. 47.50 SELECT * FROM Pedidos WHERE total_pedido(codigoPedido) > 100; ← usable in WHERE too

SELECT … INTO variable

This is how you save a query result into a variable. Used in both functions and procedures when you need a single value from a table.

SELECT SUM(cantidad * precioUnidad) INTO total ← saves here FROM DetallePedidos WHERE codigoPedido = 1;

IFNULL(x, 0)

If the value is NULL (e.g. a client with no orders), use 0 instead of NULL. NULL would break any math calculation.

-- Without IFNULL: total = NULL → error! -- With IFNULL: IFNULL(NULL, 0) → 0 ✅ IFNULL(47.50, 0) → 47.50 ✅
