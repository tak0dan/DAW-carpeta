1. How to use this guide

Read it like a field manual, not like a blog post

If you feel lost in JavaScript, do not start by memorizing syntax in isolation. Start by understanding the execution story. The browser loads HTML. JavaScript runs after or during page load. JavaScript can read elements, react to clicks, change text, store data, ask APIs for information, wait for asynchronous results, and then update the screen. That story is the spine of most beginner and intermediate browser exercises. Once you understand the story, every chapter in this guide becomes easier because you know where the concept fits.

Use the guide in layers. First read the summary paragraphs and examples. Second copy the code snippets and run them in a small test file. Third solve one micro-task with the same pattern. Fourth come back and read the common mistakes section for that topic. That loop is much more effective than passive reading because JavaScript is learned by seeing values move through the code. Even the simplest example becomes clearer once you actually change a value, press a button, inspect the result, and then trace the flow line by line.

  • Read one chapter summary.
  • Study the example code.
  • Rewrite the example using your own variable names.
  • Add one tiny improvement.
  • Explain aloud what happens in order.
  • Keep the templates section open during practice.
On most browser-based exams the winning pattern is simple: select DOM elements, listen to an event, validate data, run logic, update the page, and handle errors visibly.
2. JavaScript mental model

What JavaScript is doing while your page is alive

This diagram is the core exam story. HTML creates the elements. JavaScript gets references to those elements with selectors like document.getElementById or querySelector. Then JavaScript registers event listeners, such as clicks, changes, submits, or key presses. When an event occurs, JavaScript runs a function. That function may do immediate work, such as adding two numbers, or delayed work, such as waiting for an API response. Finally, JavaScript writes the result back into the page by changing text, HTML, styles, classes, or created elements. If you can narrate that sequence, you can explain most beginner browser exercises correctly.

ABasic DOM + click template

This is the default browser-exercise skeleton. Use it when the task says "take a value from the form and show a result." It covers DOM selection, click handling, input reading, validation, and DOM update in the smallest useful shape. 

<input type="text" id="dato">
<button id="boton">Go</button>
<div id="resultado"></div>

<script>
const dato = document.getElementById("dato");
const boton = document.getElementById("boton");
const resultado = document.getElementById("resultado");

boton.addEventListener("click", () => {
  const valor = dato.value.trim();

  if (valor === "") {
    resultado.textContent = "Introduce un valor";
    return;
  }

  resultado.textContent = "Has escrito: " + valor;
});
</script>

BForm submit template

Use this when the HTML uses a real form. The critical step is event.preventDefault() so the browser does not reload the page. In an exam, mentioning why you prevent the default submit behavior shows good understanding of browser flow. 

const form = document.getElementById("form");
const input = document.getElementById("name");
const resultado = document.getElementById("resultado");

form.addEventListener("submit", (event) => {
  event.preventDefault();

  const nombre = input.value.trim();

  if (nombre === "") {
    resultado.textContent = "Escribe un nombre";
    return;
  }

  resultado.textContent = "Hola " + nombre;
});

LDOM creation with helper + container template

Use this when the result is not one line but a group of repeated lines or cards. The important design idea is that one helper creates one visual block, and the main render function only loops and appends. This keeps the code readable even when the amount of output grows. 

function crearTarjetaUsuario(usuario) {

7. Snippet composition lab

How to build bigger JavaScript answers from small reliable blocks

This section exists for one reason: in real coding exams you are rarely rewarded for knowing one giant perfect solution by memory. You are rewarded for being able to assemble a correct answer from smaller parts without losing the flow. That means you should think of JavaScript solutions as stacks of blocks. One block talks to HTML. One block reads values. One block converts values. One block validates values. One block calculates or fetches. One block renders. One block handles error. One block sometimes stores data. Once you see the program as a composition of blocks, advanced exercises stop feeling magical. They become ordered assembly.

The easiest mistake beginners make is trying to jump directly from the task statement to the final full app. That feels efficient, but it usually creates confusion because several responsibilities are mixed at once. Instead, solve the exercise in layers. First make HTML and JavaScript communicate. Then prove you can read one value. Then prove you can validate it. Then prove you can render a fixed result. Then replace the fixed result with real logic. Then add the asynchronous or storage layer if the task needs it. This order is slower for thirty seconds and faster for the rest of the exam.

Another key idea is that the same small blocks can be reused in many different exercises. A value conversion block is useful in calculators, converters, weather lookups with numeric IDs, or any form that expects numbers. A render-list block is useful in user fetchers, currency lists, crypto lists, and character generators. An error-first validation block is useful almost everywhere. This is why a snippet handbook is so powerful: it teaches transferable units, not isolated answers.

The composition strategy also makes debugging easier. If the page fails, you can ask which block is broken. Did HTML and JS connect correctly. Did the event fire. Did the conversion fail. Did the helper return the wrong value. Did the render block point at the wrong element. Breaking the answer into blocks creates natural checkpoints. In exam conditions, checkpoints are more valuable than elegance. They give you a controlled way to recover.

Lab 1 - How JavaScript communicates with HTML

JavaScript cannot modify HTML by guessing. It needs references to actual DOM elements. This is why IDs, selectors, and DOM references are always near the start of small browser scripts. The HTML gives names to the important elements, and JavaScript uses those names to find the elements later. This is the first communication bridge between markup and code. If this bridge fails, everything else fails too.

The simplest communication pattern is: give the HTML element an id, then select it in JavaScript with document.getElementById. Once you have the element, you can read its value, change its text, attach an event listener, or append children into it. In beginner exercises, that is enough. You do not need a framework to understand browser interaction. You need the DOM bridge to be clear.

Think of HTML as the visible shell and JavaScript as the behavior layer. HTML says "this is the input, this is the button, this is the result area." JavaScript says "when the button is clicked, read the input and write to the result area." That is the actual conversation between the two technologies. 

<input type="text" id="nombre">
<button id="saludar">Saludar</button>
<p id="resultado"></p>

<script>
const nombre = document.getElementById("nombre");
const saludar = document.getElementById("saludar");
const resultado = document.getElementById("resultado");

saludar.addEventListener("click", () => {
  resultado.textContent = "Hola " + nombre.value;
});
</script>

Notice the chain. HTML defines ids. JavaScript selects ids. The event starts the code. The code reads nombre.value. Then it writes resultado.textContent. If you understand this chain, you understand the core communication mechanism of the browser for most beginner tasks.

Lab 2 - Building advanced code from simple snippets

Advanced code is often just simple code stacked in the right order. Start with one fixed DOM update. Then add input reading. Then add validation. Then add helper functions. Then add async work if needed. Each stage is small, but together they become a real app. The mistake is to treat the final app as a single mysterious unit. It is not. It is only a stack of snippets that now cooperate.

This is one of the best study methods in the whole file. Take a tiny working snippet and improve it in layers. Each layer should solve one clear weakness of the previous layer. That is exactly how real app growth happens, and it is exactly how you should train for a code exam.

Lab 3 - Conversion and validation as a pipeline

Many exam answers fail because values are used too early. A number should not be calculated with until it has been read, cleaned, converted, and validated. Treat these as separate stages. Read the raw text. Clean spaces. Convert to number. Check for NaN. Check the allowed range. Only then continue. If you do not separate these steps mentally, debugging becomes much harder because you do not know which stage broke.

Stage Question Typical code
Read What exactly came from the input? const texto = input.value.trim();
Convert Can this become a number? const numero = parseFloat(texto);
Validate Is the number acceptable? if (isNaN(numero) || numero <= 0)
Use Now can I calculate or request safely? const total = numero * cambio; 
function obtenerCantidadValida(input) {
  const texto = input.value.trim();
  const numero = parseFloat(texto);

  if (texto === "") return null;
  if (isNaN(numero)) return null;
  if (numero <= 0) return null;

  return numero;
}

This pipeline is reusable in calculators, converters, API IDs, random-range inputs, and many form tasks. The more automatic this flow becomes in your head, the fewer careless bugs you will make.

Lab 4 - Error-aware composition

Strong code is not code that only handles the happy path. Strong code also decides what should happen when the input is missing, the conversion fails, the response is not OK, or the JSON shape is not usable. The easiest way to keep this under control is to place an error decision after each risky stage. This gives the script several safe exits instead of one uncontrolled crash. 

boton.addEventListener("click", async () => {
  const ciudad = inputCiudad.value.trim();

  if (ciudad === "") {
    resultado.textContent = "Escribe una ciudad";
    return;
  }

  resultado.textContent = "Cargando...";

  try {
    const response = await fetch(urlBase + ciudad);

    if (!response.ok) {
      throw new Error("HTTP " + response.status);
    }

    const data = await response.json();

    if (!data.main || !data.weather || !data.weather[0]) {
      throw new Error("Respuesta incompleta");
    }

    resultado.textContent = data.main.temp + " C - " + data.weather[0].description;
  } catch (error) {
    console.error(error.message);
    resultado.textContent = "No se pudo completar la operacion";
  }
});

The point of this snippet is not only that it catches errors. The point is that the risky places are visible: empty input, bad HTTP response, incomplete JSON shape, final fallback message. This kind of clarity makes an exam answer feel dependable.

Lab 5 - Draw the code before you write the code

Sometimes a problem becomes much easier when you sketch the structure first. You do not need art. You need sequence. If the task is "fetch data and show it in a list," the sketch can simply be HTML input -> click event -> helper fetch -> array loop -> create li -> append li. This is enough to guide implementation.

This kind of diagram is more practical than it looks. It forces you to identify the pieces before syntax noise appears. Under exam pressure, that is extremely valuable.

Lab 7 - How one helper can feed another helper

A very common source of confusion is the moment when one helper function must pass its result into another helper function. Beginners sometimes feel that this is already "advanced" code, but it is actually just the natural next step after learning returns. One helper reads. Another validates. Another transforms. Another renders. The output of one block becomes the input of the next block. That is not complication. That is the normal shape of a maintainable program.

The important mental model is to treat functions like stages in a machine. The first stage cleans the raw value. The second stage checks whether the value is acceptable. The third stage transforms the value into the business result. The fourth stage turns the business result into visible output. Once you imagine the program in stages, function nesting becomes much easier to read and explain. You are not "calling functions inside functions" randomly. You are moving data from one named stage to the next.

This is also one of the best ways to create advanced-looking answers from simple code. Each helper can stay short and simple. The sophistication comes from composition, not from complex syntax inside any one function. That is a reassuring idea for exam conditions, because it means you do not need to invent genius code. You need to connect reliable pieces cleanly. 

function leerTexto(input) {
  return input.value.trim();
}

function convertirANumero(texto) {
  return parseFloat(texto);
}

function numeroEsValido(numero) {
  return !isNaN(numero) && numero > 0;
}

function calcularIva(numero) {
  return numero * 1.21;
}

function pintarTotal(total) {
  resultado.textContent = "Total con IVA: " + total;
}

boton.addEventListener("click", () => {
  const texto = leerTexto(inputPrecio);
  const numero = convertirANumero(texto);

  if (!numeroEsValido(numero)) {
    resultado.textContent = "Precio no valido";
    return;
  }

  const total = calcularIva(numero);
  pintarTotal(total);
});

Notice how the event handler reads almost like a sentence. That is a good sign. Advanced code is often code that reads clearly from top to bottom because every step has already been named.

Lab 8 - Multi-input assembly: several HTML fields, one clear flow

Many students feel safe with one input field but become uncertain as soon as the form contains two or three fields. The solution is not to invent a whole new architecture. The same composition rule still works. Read each field. Convert the ones that need conversion. Validate each rule. Combine the accepted values into one object or one final calculation. Then render the result. The presence of multiple inputs increases the count of checks, but not the nature of the solution.

It is often helpful to create one helper that gathers all current input values into a single object. Then the later helpers operate on that object instead of pulling directly from the DOM again and again. This keeps the later logic cleaner because it is working with data, not with HTML elements. It also makes the answer easier to explain because you can say that the first stage extracts a normalized data object from the form.

Another benefit of this pattern is that validation messages become easier to control. You can validate one property at a time and stop at the first failing rule, or you can collect several errors if the task wants that. The structure is flexible because the read stage and the validate stage are already separated. 

function leerFormulario() {
  return {
    nombre: inputNombre.value.trim(),
    edad: parseInt(inputEdad.value),
    ciudad: inputCiudad.value.trim()
  };
}

function validarFormulario(datos) {
  if (datos.nombre === "") return "Nombre obligatorio";
  if (isNaN(datos.edad) || datos.edad <= 0) return "Edad no valida";
  if (datos.ciudad === "") return "Ciudad obligatoria";
  return null;
}

form.addEventListener("submit", (event) => {
  event.preventDefault();
  const datos = leerFormulario();
  const error = validarFormulario(datos);

  if (error) {
    resultado.textContent = error;
    return;
  }

  resultado.textContent = datos.nombre + " - " + datos.edad + " - " + datos.ciudad;
});

This is still the same exam pattern you already know. It simply proves that the pattern scales cleanly when more than one field participates.

Lab 9 - Error matrix: what to write for each problem type

When coding fast, it helps to know the exact fallback message and code shape you will use for each major problem category. That way you do not waste time inventing error behavior from zero. Instead, you choose from a small error matrix. Input error. Conversion error. Validation error. HTTP error. Empty storage. Missing JSON field. Each category can have one standard response pattern that you reuse confidently.

Problem type Typical test Typical response
Empty text texto === "" Show "Introduce un valor"
Bad number isNaN(numero) Show "Numero no valido"
Bad range numero <= 0 Show "Debe ser mayor que cero"
HTTP failure !response.ok Throw error and show fallback text
Missing stored data !guardado Show "No hay datos guardados"
Unexpected JSON shape !data.main Throw or render "Respuesta incompleta"

This is not about making error handling robotic. It is about making it reliable. In exam settings, consistency is a strength. When you know the fallback structure in advance, you preserve more time for the logic that is actually unique to the task.

Lab 10 - From several simple snippets to a mini dashboard

The final composition jump is moving from a one-task page to a page with several small modules. This may sound more complex, but again the same logic applies. Each module has its own trigger, input, validation, logic, and render block. The page simply contains several such flows side by side or stacked one after another. If you keep the modules isolated, the page remains understandable even though it has more features.

Imagine a page that greets a saved user, fetches one current weather record, and calculates a conversion. You do not need one giant universal function. You need three small flows sharing one file. One localStorage flow. One fetch flow. One converter flow. They may reuse some helper ideas, but each module still has a recognizable internal pipeline. This is how real single-file projects grow without collapsing into confusion.

In certification contexts, even if the exam does not ask for a full dashboard, thinking modularly like this can still improve a smaller answer. It reminds you that features should be grouped by responsibility rather than by the order in which you happened to type them. 

// module 1: greeting
function cargarNombreGuardado() {
  const guardado = localStorage.getItem("nombre");
  saludo.textContent = guardado ? "Hola " + guardado : "Hola";
}

// module 2: converter
function calcularTotal(cantidad, cambio) {
  return Math.round(cantidad * cambio * 100) / 100;
}

// module 3: weather
async function pedirTiempo(ciudad) {
  const response = await fetch(apiBase + ciudad);
  if (!response.ok) throw new Error("Error HTTP");
  return await response.json();
}

What matters here is not the specific features. What matters is the growth pattern: small reliable snippets can coexist and build a larger application without changing the fundamental model you already know.

8. Micro examples atlas

One by one: tiny JavaScript actions explained as separate building blocks

This section is intentionally granular. Each box answers a very small practical question. How do I show text. How do I show only part of the text. How do I concatenate values. How do I do a simple equation. How do I create a new button. How do I place that button above or below something. How do I hide or remove part of the screen. How do I move through an array one value at a time. How do I generate buttons from array data. If the earlier sections teach structure, this section teaches tiny moves. Small moves are what advanced answers are made from.

Read these examples slowly and literally. Do not worry about memorizing all of them at once. The goal is to train your sense that browser behavior is made from many tiny reversible steps. You select an element. You read a value. You write a property. You create a node. You append a node. You remove a node. You add a class. You loop over an array. Once these actions feel ordinary, the larger tasks stop feeling impossible.

Example 1 - Show fixed text when a button is clicked

This is the smallest visible behavior in browser JavaScript. A click happens, and the page changes. Use it as the mental starting point for more complex interactions. 

<button id="boton">Mostrar</button>
<p id="resultado"></p>

const boton = document.getElementById("boton");
const resultado = document.getElementById("resultado");

boton.addEventListener("click", () => {
  resultado.textContent = "Hola mundo";
});

The key line is resultado.textContent = "Hola mundo". That is the simplest possible render action.

Example 2 - Show text taken from an input field

Now the result is not fixed. It comes from HTML input. This is where JavaScript stops being only output and starts becoming input plus output. 

<input id="nombre" type="text">
<button id="saludar">Saludar</button>
<p id="resultado"></p>

const nombre = document.getElementById("nombre");
const saludar = document.getElementById("saludar");
const resultado = document.getElementById("resultado");

saludar.addEventListener("click", () => {
  resultado.textContent = "Hola " + nombre.value;
});

The bridge is nombre.value. That is how JavaScript reads what the user typed into HTML.

Example 3 - Show only one part of the text

Sometimes you do not want the whole string. You only want part of it. A very practical tool is slice. It cuts the string between two positions. 

const texto = "JavaScript";
const parte = texto.slice(0, 4);
console.log(parte); // Java

resultado.textContent = parte;

If the task says "show the first 3 letters" or "show only the first word", think about string tools such as slice, split, and trim.

Example 4 - Show the second word of a sentence

If the text contains several words, a simple trick is to split the sentence into an array of words and then choose the one you want. 

const frase = "cat dog horse";
const palabras = frase.split(" ");
const segunda = palabras[1];
resultado.textContent = segunda; // dog

This is a good example of how string work and array work often connect directly.

Example 5 - Concatenate text and numbers into one message

Concatenation means joining pieces into one longer string. It is one of the most frequent operations in small browser apps because the UI often needs a sentence built from several values. 

const nombreUsuario = "Ana";
const edad = 22;
resultado.textContent = "Usuario: " + nombreUsuario + " - Edad: " + edad;

You can also use template literals if they feel clearer. 

resultado.textContent = \`Usuario: \${nombreUsuario} - Edad: \${edad}\`;

Example 6 - Do a simple equation from two inputs

The important part is conversion. Input values arrive as strings, so you must convert them before adding. 

const a = parseFloat(inputA.value);
const b = parseFloat(inputB.value);

if (isNaN(a) || isNaN(b)) {
  resultado.textContent = "Escribe dos numeros";
  return;
}

resultado.textContent = a + b;

Without conversion, "7" + "9" becomes "79".

Example 7 - Do a more complicated equation

When the formula gets larger, it helps to store intermediate results in variables. That makes the math and the explanation easier. 

const base = parseFloat(inputBase.value);
const altura = parseFloat(inputAltura.value);

if (isNaN(base) || isNaN(altura)) {
  resultado.textContent = "Datos no validos";
  return;
}

const area = (base * altura) / 2;
resultado.textContent = "Area: " + area;

Intermediate variables like area make a solution look cleaner and easier to debug.

Example 8 - Create a new button with JavaScript

Here JavaScript does not only update existing HTML. It creates a brand-new element from nothing. 

const nuevoBoton = document.createElement("button");
nuevoBoton.textContent = "Nuevo boton";
document.body.appendChild(nuevoBoton);

This is the base pattern for dynamic element creation: create, fill, append.

Example 9 - Create a button below another element

"Below" usually means append the new element inside a parent after the existing children. 

const zona = document.getElementById("zona");
const boton = document.createElement("button");
boton.textContent = "Debajo";
zona.appendChild(boton);

If zona already contains text or another button, the new button appears after them, so visually it will usually be below if the container is vertical.

Example 10 - Create a button above another element

To place a new element before the existing first child, use prepend. 

const zona = document.getElementById("zona");
const boton = document.createElement("button");
boton.textContent = "Encima";
zona.prepend(boton);

The difference between appendChild and prepend is fundamental for placement.

Example 11 - Create a button on the right or on the left

JavaScript creates the element, but CSS controls whether the element sits left or right. The usual trick is to place both buttons inside a flex container and choose the order or alignment. 

<div id="fila" style="display:flex; gap:10px;"></div>

const fila = document.getElementById("fila");
const izquierda = document.createElement("button");
const derecha = document.createElement("button");

izquierda.textContent = "Izquierda";
derecha.textContent = "Derecha";

fila.appendChild(izquierda);
fila.appendChild(derecha);

The first appended button appears first in the row. That usually means left. The second usually means right.

Example 12 - Hide part of the window on click

Hiding is usually changing the style or a class of an existing element. The fastest beginner method is style.display = "none". 

const caja = document.getElementById("caja");
const cerrar = document.getElementById("cerrar");

cerrar.addEventListener("click", () => {
  caja.style.display = "none";
});

The element still exists in the DOM, but it disappears visually.

Example 13 - Show it again after hiding it

If one button hides and another shows, the box becomes a simple visibility-state exercise. 

ocultar.addEventListener("click", () => {
  caja.style.display = "none";
});

mostrar.addEventListener("click", () => {
  caja.style.display = "block";
});

The exact display value may be block, flex, or something else depending on layout.

Example 14 - Toggle hide/show with one button

A toggle is useful when one control must alternate between two states. 

botonToggle.addEventListener("click", () => {
  if (caja.style.display === "none") {
    caja.style.display = "block";
  } else {
    caja.style.display = "none";
  }
});

This is the simplest state toggle pattern with DOM styles.

Example 15 - Remove an element completely

Hiding and removing are different. Removing deletes the element from the DOM tree. 

const aviso = document.getElementById("aviso");
aviso.remove();

After remove(), the node is gone, not only invisible.

Example 16 - Show the next item of an array each time you click

This pattern uses an index variable that stays in outer scope. Each click moves to the next element. 

const animales = ["cat", "dog", "horse"];
let indice = 0;

boton.addEventListener("click", () => {
  resultado.textContent = animales[indice];
  indice++;

  if (indice >= animales.length) {
    indice = 0;
  }
});

This is a classic example of state that survives between clicks.

Example 17 - Create one button for each word in an array

This directly answers the "cat, dog, horse" style task. The array drives repeated button creation. 

const palabras = ["cat", "dog", "horse"];
const zona = document.getElementById("zona");

palabras.forEach((palabra) => {
  const boton = document.createElement("button");
  boton.textContent = palabra;
  zona.appendChild(boton);
});

The loop is the key. One array item becomes one button.

Example 18 - Create those buttons only after a click

The array-driven generation can be delayed until the user requests it. 

crearBotones.addEventListener("click", () => {
  palabras.forEach((palabra) => {
    const boton = document.createElement("button");
    boton.textContent = palabra;
    zona.appendChild(boton);
  });
});

If you want to avoid duplicates, clear the container first with zona.innerHTML = "".

Example 19 - Add a fourth word, then create another row of buttons

This matches the idea of producing one set first and then another larger set below it. The trick is to create a row container every time the click happens, then fill that row from the current array contents. 

const palabras = ["cat", "dog", "horse"];
const zona = document.getElementById("zona");

function crearFilaBotones() {
  const fila = document.createElement("div");
  fila.style.marginTop = "10px";

  palabras.forEach((palabra) => {
    const boton = document.createElement("button");
    boton.textContent = palabra;
    fila.appendChild(boton);
  });

  zona.appendChild(fila);
}

primerClick.addEventListener("click", () => {
  crearFilaBotones();
});

segundoClick.addEventListener("click", () => {
  palabras.push("lion");
  crearFilaBotones();
});

The key idea is that the data array changed before the second row was generated, so the second row reflects the new state of the array.

Example 20 - Create buttons from whatever the user typed

Now the array is not fixed in code. The user types comma-separated words, and JavaScript turns them into buttons. 

botonCrear.addEventListener("click", () => {
  const texto = inputPalabras.value.trim();
  const palabras = texto.split(",");

  zona.innerHTML = "";

  palabras.forEach((palabra) => {
    const boton = document.createElement("button");
    boton.textContent = palabra.trim();
    zona.appendChild(boton);
  });
});

This example joins text processing, arrays, loops, trimming, and DOM creation in one small task.

Example 21 - Put created buttons into separate rows

If the exam asks for groups below each other, do not append all buttons directly to the same parent. Create a new wrapper div for each group. 

function crearFilaConBotones(listaPalabras) {
  const fila = document.createElement("div");
  fila.style.display = "flex";
  fila.style.gap = "8px";
  fila.style.marginTop = "8px";

  listaPalabras.forEach((palabra) => {
    const boton = document.createElement("button");
    boton.textContent = palabra;
    fila.appendChild(boton);
  });

  zona.appendChild(fila);
}

Wrapper elements are the secret for layout control.

Example 22 - Make one created button delete itself on click

Created elements can also receive their own event listeners. 

const boton = document.createElement("button");
boton.textContent = "Borrame";

boton.addEventListener("click", () => {
  boton.remove();
});

zona.appendChild(boton);

This is an important step because it proves that dynamic elements can also be interactive, not only static.

Example 23 - Make one click hide a different element

A button does not have to change itself. It can target any other DOM node. 

botonCerrar.addEventListener("click", () => {
  panelInfo.style.display = "none";
});

This is the normal basis for dismissible alerts, popups, side panels, or help boxes.

Example 24 - Spawn a new button to the left of an old button

If two buttons share a parent row, the left or right position depends on insertion order. 

const fila = document.getElementById("fila");
const botonViejo = document.getElementById("viejo");
const nuevo = document.createElement("button");
nuevo.textContent = "Nuevo a la izquierda";

fila.insertBefore(nuevo, botonViejo);

insertBefore is a very useful DOM tool whenever placement relative to a specific sibling matters.

Example 25 - Spawn a new button to the right of an old button

If you want a new node after an existing one, you can use after on that existing node. 

const nuevo = document.createElement("button");
nuevo.textContent = "Nuevo a la derecha";
botonViejo.after(nuevo);

This is the sibling-based version of placement.

Example 26 - Build a disappearing message box

Many practical interfaces need a box that can be closed. Here is the smallest pattern. 

<div id="mensaje">
  Operacion completada
  <button id="cerrarMensaje">X</button>
</div>

cerrarMensaje.addEventListener("click", () => {
  mensaje.remove();
});

This pattern appears in alerts, notifications, popups, and dismissible status messages.

Example 27 - Show one equation and the equation result together

Sometimes the UI should show both the formula and the result. That is still just string construction. 

const a = 7;
const b = 9;
const total = a + b;

resultado.textContent = a + " + " + b + " = " + total;

This is a small but useful presentation trick in school-style math tools.

Example 28 - Show only the newest generated group

If the task wants replacement instead of accumulation, clear the container first. 

botonGenerar.addEventListener("click", () => {
  zona.innerHTML = "";
  crearFilaConBotones(palabras);
});

Remember this contrast: innerHTML = "" means replace; no clearing means accumulate.

Example 29 - Accumulate groups one below another

If the task wants history, do not clear the container. Append each new row and let the old rows remain. 

botonGenerar.addEventListener("click", () => {
  crearFilaConBotones(palabras);
});

This one-line difference changes the behavior of the page completely.

Example 30 - Grow one tiny example into a mini app

Here is the full growth ladder. A single button first shows fixed text, then dynamic text, then validated text, then created buttons, then repeated groups. This is the exact mindset you should carry into certification work: complexity is layered, not summoned instantly. 

// level 1
resultado.textContent = "Hola";

// level 2
resultado.textContent = input.value;

// level 3
const texto = input.value.trim();
if (texto === "") return;
resultado.textContent = texto;

// level 4
const boton = document.createElement("button");
boton.textContent = texto;
zona.appendChild(boton);

// level 5
const palabras = texto.split(",");
crearFilaConBotones(palabras.map((p) => p.trim()));

The lesson is simple: advanced JavaScript is often only beginner JavaScript repeated with better structure.

9. 15-part progression

Your requested practical ladder, written explicitly as categories

This section mirrors your exact progression. The rest of the handbook already contains the code blocks, but this chapter makes the roadmap explicit so you can study in order and know what each stage is supposed to teach. Think of it as the table of exercises you could spend months tinkering with from zero up to a certification-ready level.

1. Text & Equations

Idea: Manipulate text content, show math results, and update them dynamically.

Start here because it teaches the most basic browser contract: read something, compute something, show something. At first, just print fixed text. Then print text from an input. Then print one equation. Then print a more complex formula using intermediate variables. Then add a timer that updates the result over time. 

// fixed text
resultado.textContent = "Hola";

// simple equation
resultado.textContent = 5 + 7;

// equation from inputs
const a = parseFloat(inputA.value);
const b = parseFloat(inputB.value);
const c = parseFloat(inputC.value);
const d = parseFloat(inputD.value);
const e = parseFloat(inputE.value);
const formula = (a * b + c ** 2) / (d - e);
resultado.textContent = formula;

// live update
setInterval(() => {
  reloj.textContent = new Date().toLocaleTimeString();
}, 1000);

2. Creating, Positioning, and Removing Elements

Idea: Programmatically create and place elements anywhere.

The key DOM actions here are createElement, appendChild, prepend, insertBefore, after, and remove. Once you understand these, the page stops being fixed HTML and becomes a canvas you can build while the user interacts with it. 

const nuevo = document.createElement("button");
nuevo.textContent = "Nuevo";
contenedor.appendChild(nuevo);   // below / after existing children
contenedor.prepend(nuevo);       // above / before existing children
botonViejo.after(nuevo);         // right or after specific sibling
botonViejo.before(nuevo);        // left or before specific sibling
nuevo.remove();                  // remove completely

3. Visibility Control

Idea: Show, hide, collapse, and toggle parts of the UI.

This stage teaches state. The same element can be visible, hidden, collapsed, faded, or toggled depending on user action. These are not advanced ideas; they are just state changes reflected visually. Learn both direct style control and class-based control so you can choose the right level of structure. 

// hide itself
boton.addEventListener("click", () => {
  boton.style.display = "none";
});

// A hides B
botonA.addEventListener("click", () => {
  panelB.style.display = "none";
});

// toggle
botonToggle.addEventListener("click", () => {
  panel.classList.toggle("oculto");
});

4. Arrays & Generated UI

Idea: Turn array values into visible interface elements.

This stage teaches repetition from data. One button can cycle through an array. One click can create one button per array item. Adding new array values changes the next generated UI. This is the moment where data and DOM start feeling deeply connected. 

const palabras = ["cat", "dog", "horse"];

// cycle
let indice = 0;
siguiente.addEventListener("click", () => {
  resultado.textContent = palabras[indice];
  indice = (indice + 1) % palabras.length;
});

// generate buttons
crear.addEventListener("click", () => {
  palabras.forEach((palabra) => {
    const boton = document.createElement("button");
    boton.textContent = palabra;
    zona.appendChild(boton);
  });
});

5. Panels, Tabs, Accordion

Idea: Generate structured UI sections.

Once you can create elements and hide/show them, you can build panels, tabs, and accordions. The logic is still the same: create containers, store references, and control which one is currently active. These are good exercises because they teach organization of several related elements at once. 

const tabs = ["Info", "Stats", "Settings"];

tabs.forEach((nombre) => {
  const boton = document.createElement("button");
  const panel = document.createElement("div");

  boton.textContent = nombre;
  panel.textContent = "Contenido de " + nombre;
  panel.style.display = "none";

  boton.addEventListener("click", () => {
    document.querySelectorAll(".panel").forEach((p) => p.style.display = "none");
    panel.style.display = "block";
  });
});

6. Button Armies

Idea: Buttons that generate more buttons, sometimes recursively.

This is where UI creation becomes playful and a little absurd. One button can create ten more. One of those can create children. A grid can be generated from a number input. These exercises are excellent for training loops, dynamic creation, event attachment on created elements, and wrapper-container layout. 

crearDiez.addEventListener("click", () => {
  for (let i = 0; i < 10; i++) {
    const boton = document.createElement("button");
    boton.textContent = "Boton " + (i + 1);
    ejercito.appendChild(boton);
  }
});

7. Event Handling Mastery

Idea: Total control over event flow.

At this point you stop seeing events as only click handlers. You now study hover, keypress, bubbling, capturing, and event delegation. Event delegation is especially powerful because one parent can manage many dynamic children, which is perfect for large generated interfaces such as button armies or dynamic lists. 

lista.addEventListener("click", (event) => {
  if (event.target.matches("button")) {
    console.log("Se hizo click en", event.target.textContent);
  }
});

8. Layout & Position Control

Idea: Programmatically alter where and how things appear.

This stage is about using JavaScript to change layout rules, not only content. You can switch flex direction, resize elements, or respond to drag-like interactions. The important lesson is that JavaScript can change both meaning and placement. UI is not only text. UI is also geometry. 

cambiarDireccion.addEventListener("click", () => {
  contenedor.style.display = "flex";
  contenedor.style.flexDirection =
    contenedor.style.flexDirection === "row" ? "column" : "row";
});

9. Forms + Validation

Idea: Read, validate, and dynamically update forms.

This is the bridge from small demos to serious browser tasks. Form validation, live preview, disabled submit, and dynamic fields all belong here. This stage teaches that good apps do not only read values. They defend the quality of those values before using them. 

form.addEventListener("input", () => {
  const nombre = inputNombre.value.trim();
  preview.textContent = "Preview: " + nombre;
  submit.disabled = nombre === "";
});

10. Data Structures → UI

Idea: Build UI from arrays of objects and nested JSON-like data.

This is where data shape becomes central. Arrays of objects become tables. JSON becomes menus. Datasets become panels. Once you are comfortable with this stage, APIs become much easier because you already know how to move structured data into visible structures. 

const usuarios = [
  { nombre: "Ana", edad: 22 },
  { nombre: "Luis", edad: 31 }
];

usuarios.forEach((usuario) => {
  const fila = document.createElement("tr");
  fila.innerHTML = \`\${usuario.nombre}\${usuario.edad}\`;
  tabla.appendChild(fila);
});

11. Component Systems (No Framework)

Idea: Use classes or factories to generate reusable UI components.

At this stage you stop writing only one-off snippets. You start writing little reusable factories or classes that know how to create themselves, update themselves, and destroy themselves. This is not a framework. It is disciplined reuse. It is how advanced vanilla JavaScript starts to feel architectural. 

class SimpleButton {
  constructor(texto) {
    this.element = document.createElement("button");
    this.element.textContent = texto;
  }

  mount(parent) {
    parent.appendChild(this.element);
  }

  destroy() {
    this.element.remove();
  }
}

12. Animation & Interaction

Idea: Add motion triggered by JavaScript.

Motion usually means JavaScript changes classes or styles and CSS performs the animation. This stage teaches that interaction is not only data and structure. It is also timing, transition, and feedback. Even a simple fade or sliding panel can make state changes clearer. 

botonAnimar.addEventListener("click", () => {
  panel.classList.add("fade-in");
});

13. Mini-Apps

Idea: Combine everything into functional little systems.

Here you stop practicing only isolated tricks and start assembling complete but still small tools: counter, todo list, notes app, preview form, weather loader, converter, or dynamic tabbed panel. Mini-apps are where structure becomes real because several blocks now cooperate in one flow. 

// counter mini-app idea
let total = 0;
mas.addEventListener("click", () => {
  total++;
  salida.textContent = total;
});
menos.addEventListener("click", () => {
  total--;
  salida.textContent = total;
});
reset.addEventListener("click", () => {
  total = 0;
  salida.textContent = total;
});

14. Absurd UI Logic

Idea: Inputs define how many times a button acts, how many elements spawn, and how everything interacts.

This is a fantastic training stage because it forces you to think compositionally. An input can define repeat count. Another can define spawn count. Another can define the text to print. One button can trigger a cascade of repeated actions. These exercises look silly, but they are brilliant for mastering loops, state, DOM generation, and event flow. 

botonLanzar.addEventListener("click", () => {
  const texto = inputTexto.value.trim();
  const repeticiones = parseInt(inputRepeticiones.value);
  const cantidadBotones = parseInt(inputBotones.value);

  for (let i = 0; i < repeticiones; i++) {
    const linea = document.createElement("p");
    linea.textContent = texto;
    zona.appendChild(linea);
  }

  for (let i = 0; i < cantidadBotones; i++) {
    const boton = document.createElement("button");
    boton.textContent = "Spawn " + (i + 1);
    zona.appendChild(boton);
  }
});

15. Full Chaos Mode

Idea: Several inputs define several behavior dimensions at once.

This is the extreme composition stage. Input one defines repeats. Input two defines spawned elements. Input three defines delay. Input four defines type. Input five defines whether the new things can be removed. One master button reads all of them and produces a controlled UI cascade. This stage is not about building the prettiest app. It is about proving that you understand how small JavaScript blocks can be combined into a rich dynamic system. 

masterButton.addEventListener("click", async () => {
  const repeticiones = parseInt(inputRepeticiones.value);
  const cantidad = parseInt(inputCantidad.value);
  const delay = parseInt(inputDelay.value);
  const tipo = inputTipo.value;
  const removible = checkRemovible.checked;

  for (let r = 0; r < repeticiones; r++) {
    for (let i = 0; i < cantidad; i++) {
      let elemento;

      if (tipo === "button") elemento = document.createElement("button");
      if (tipo === "panel") elemento = document.createElement("div");
      if (tipo === "text") elemento = document.createElement("p");

      elemento.textContent = tipo + " " + (i + 1);

      if (removible) {
        elemento.addEventListener("click", () => elemento.remove());
      }

      zona.appendChild(elemento);
    }

    await new Promise((resolve) => setTimeout(resolve, delay));
  }
});

This is still built from the same simple parts you learned earlier: input reading, conversion, loops, creation, event listeners, conditionals, and timed waiting. The complexity is real, but it is assembled complexity.

10. Exam strategy

How to answer JavaScript tasks under pressure

Universal exam algorithm

  1. Read the exercise and identify input, process, and output.
  2. Write the HTML references you need: inputs, button, result area.
  3. Select the DOM elements in JavaScript.
  4. Attach the correct event listener.
  5. Read and validate user data.
  6. Move the logic into helper functions if the handler gets long.
  7. If remote data is needed, show loading text before fetch.
  8. Handle errors explicitly.
  9. Update the page with the final result.
  10. Test the happy path and one bad path.
In written explanations, use the verbs: select, listen, read, validate, process, await, render, and handle errors. These verbs make your answer sound structured and correct.

How to explain code in words

Many students can write something that works but cannot explain it. In certification contexts, explanation is often what proves that the solution was deliberate. A reliable structure is: first say what the variables store, then say which event starts the flow, then say how the input is validated, then say which function performs the main logic, then say how the DOM is updated, and finally say how errors or edge cases are handled. That explanation format works for calculations, storage, random generators, promises, and fetch tasks.

Example exam language: "I select the input, button, and result area from the DOM. Then I add a click listener to the button. Inside that function I read the input value, convert it to a number, and validate it. If the data is invalid I show an error message and stop the execution with return. If the data is valid I call a helper function to perform the calculation. Finally I print the result in the result container." This wording is generic enough to reuse in many tasks.

  • event.preventDefault()
  • parseInt(), parseFloat(), isNaN()

  • localStorage.setItem(), getItem(), removeItem()

11. Certification appendix

Oral-ready answers, quick definitions, and recovery patterns

How to answer common theory questions in plain technical English

If somebody asks "What is JavaScript?" a strong answer is: "JavaScript is the programming language that gives behavior to web pages. It can react to user events, manipulate the DOM, communicate with APIs, store data in the browser, and run both synchronous and asynchronous logic." This answer is short, accurate, and directly relevant to browser exercises. It avoids vague slogans and names concrete capabilities. If the question instead asks what JavaScript does in a browser page, you can extend the answer by describing the flow: the page loads, JavaScript selects elements, listeners wait for events, functions process data, and the DOM updates with the result.

If the question is "What is the DOM?" answer like this: "The DOM is the browser representation of the HTML page as objects that JavaScript can access and modify. With the DOM, JavaScript can read values from inputs, change text, create elements, remove elements, add classes, and react to user actions." This answer is strong because it defines the concept and lists practical uses. In certifications, a definition that includes use cases is almost always better than a definition that sounds formal but disconnected from programming tasks.

If the question is "What is an event listener?" say: "An event listener waits for a specific event, such as a click, submit, change, or key press, and runs a callback function when that event happens." Then, if needed, give a tiny example such as a button click listener. Again, the best explanation is one that connects the abstract idea to the code pattern you actually write.

If asked "What is a promise?" use this structure: "A promise represents the future result of an asynchronous operation. It starts pending and later becomes fulfilled or rejected. We handle the result with then and catch, or more readably with async and await." This is concise, correct, and enough for the level of work covered by the LM exercises.

If the question is "What is fetch?" answer: "fetch is the browser API used to make HTTP requests. It returns a promise, so after calling fetch we normally await the response, check response.ok, convert the body with response.json, and then work with the data." Notice again that this answer is procedural. In practical JavaScript questions, procedural definitions are often the most useful.

Recovery patterns when you get stuck in the middle of an exercise

One of the most valuable exam skills is recovery. Almost everyone gets stuck sometimes. The difference between a weak attempt and a strong attempt is often whether the student knows how to recover methodically. The first recovery rule is to stop trying to think about the whole problem at once. Return to the flow. What is the trigger. What is the input. What must be validated. What is the processing step. Where should the result appear. Which part is currently missing. Breaking the problem back into these steps restores control very quickly.

The second recovery rule is to replace unknown dynamic data with known fixed data temporarily. If fetch is confusing you, hardcode a sample object and prove that your render logic works first. If the DOM rendering is confusing you, log the data to the console and verify that the logic works before you touch the page. If the timer is confusing you, hardcode one visible state change before reintroducing repetition over time. In other words, isolate the broken layer instead of treating the whole application as one black box.

The third recovery rule is to simplify the UI if needed. If the task only requires that results appear, a plain textContent update is acceptable before you beautify the layout. Under exam pressure, functioning logic with clear output usually earns more than elegant styling with incomplete logic. This is especially important when the problem statement is about JavaScript behavior, not visual polish. Make it work visibly first. Improve the presentation only after the flow is stable.

The fourth recovery rule is to use the console aggressively and deliberately. Log the input value. Log the parsed value. Log inside the event callback to prove the event fired. Log the raw API data. Log before rendering. If the logs show correct values and the page still looks wrong, the problem is probably in the DOM target or render method. If the logs are already wrong, the problem is earlier in the flow. This simple branching logic can save a huge amount of time.

The fifth recovery rule is verbal. Say aloud or write a tiny comment line for the exact sentence your function should satisfy. For example: "Read amount, validate amount, fetch rates, create list, append list." Once that sentence exists, code often becomes easier because you have reduced the task to a sequence of smaller obligations. Recovery is not about luck. It is about returning to structure.

Mini glossary of words that you should be able to use correctly

Variable: a named place that stores a value. Constant: a binding declared with const that should not be reassigned. Function: a reusable block of code. Parameter: the input a function receives. Return value: the output a function gives back. Array: an ordered list of values. Object: a collection of key-value pairs. Property: one named piece of data inside an object. Event: something that happens in the browser, such as a click or submit. Callback: a function passed to another function to be executed later or in response to something.

DOM: the browser representation of the HTML document as objects. Selector: a way to obtain DOM elements. textContent: a property used to read or write plain text inside an element. innerHTML: a property used to read or write HTML markup inside an element. localStorage: browser storage for key-value string data that survives reloads. JSON: a text format for structured data. Parse: convert text into usable program data. Stringify: convert JavaScript data into JSON text.

Promise: an object representing a future result. Pending: the promise has not finished yet. Resolved or fulfilled: the promise completed successfully. Rejected: the promise failed. Async function: a function that returns a promise automatically. Await: wait inside an async function for a promise to settle. Fetch: the browser API used to make HTTP requests. Response: the HTTP result from fetch. response.ok: a quick way to know whether the HTTP status was successful.

Validation: checking whether data is acceptable before using it. Error handling: defining what the program should do when something goes wrong. Render: show program state in the page. State: the current condition of the app, such as loading, ready, success, or error. Helper function: a small function created to keep logic organized and reusable. Scope: the region of code where a variable can be accessed. Iteration: repeating work across items or across time.

You do not need to recite these words like a dictionary entry in an exam. What matters is that when you explain your solution, you use them in the correct places. Good technical language makes your explanation sound more deliberate and often reveals that you truly understand the code flow.

One final advice about terminology: use words only when you can connect them to a concrete action in your code. Saying "callback" is useful if you can point to the function passed into an event listener, a timer, or a promise handler. Saying "state" is useful if you can show where the interface changes from loading to success or error. Saying "validation" is useful if you can identify the exact condition that stops bad data. This habit prevents empty technical language and turns your explanations into evidence-based explanations. In certifications, that is often the difference between sounding memorized and sounding competent.

12. Common mistakes

The errors beginners repeat again and again

Reading input values as numbers without conversion.
HTML inputs return strings. If you do "7" + "9" you get "79", not 16. Convert first.
Forgetting return after an error message.
If the data is invalid and you keep running the rest of the function, you create secondary bugs.
Using a function but never calling it.
Writing an arrow function expression is not the same as executing it.
Using await outside an async function.
The function that contains await must be marked async.
Not checking response.ok.
Fetch resolves even for 404 or 500 responses. You still have to validate HTTP success.
Appending to old content accidentally.
Many list rendering bugs happen because innerHTML = "" was forgotten.

Debug habit to adopt immediately

Whenever the page does not behave as expected, check five things in order: was the event fired, what exact value came from the input, what type is that value, what does the function return, and what gets written to the DOM. Simple console.log() lines are not a beginner trick. They are a professional debugging habit. Use them at the entrance and exit of functions, after parsing input, and after receiving API data. Most JavaScript bugs become obvious once you inspect the actual values instead of the values you expected.
